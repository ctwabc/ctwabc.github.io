XML external entity injection

漏洞成因

有这种漏洞就是因为后端对xml没有限制，具体是什么限制后面写，所以说利用的肯定是xml的一些语法特性。

XML语法

但其实就是DTD部分

主要关注DTD

在[]里面有ENTITY js “Jo Smith”

js就是变量 Jo Smith就是变量的值

&js 就会变成Jo Smith

主要关注SYSTEM

SYSTEM 是一个强力关键字，它的核心作用就是：告诉 XML 解析器去“外包”或者“搬运”外部的资源。

也就是说它会从后面附上的地址那数据。

也就说&js会得到/etc/passwd这个文件里面的数据。

而这个xml是传给后端的，所以这个文件就是后端的文件。

很显然这个SYSTEM关键字是不安全的应该禁止

变量的两种形式

  • 通用实体

也就是我们在刚才一直用的形式，如图file就是变量

他的变量在DTD中是不会解析的

  • 参数实体

先看对变量进行定义的方法，他是通过& 变量名 的形式定义的（注意&和变量名之间有空格）

对于变量的引用是通过&变量名。

核心就是，变量的引用是在DTD里进行的。

那这个参数实体在哪里会用到呢？

SSRF 服务端请求伪造

发表于 2026-06-09 分类于 Web安全

SSRF是一种攻击模式。
我觉得就是让对方的服务器当代理，访问内网中的服务器。
正常情况下，对方内网中不让外网访问的服务器是访问不了的，但是web服务器啥的肯定可以访问，而这个web服务器是否可以畅通无阻的与内网当中的各种服务器交流呢。在可以交流的前提下，这个web服务器有ssrf漏洞的话，就可以让攻击者访问到内网中的其他服务器了。

CSRF 跨站请求伪造详解

发表于 2026-06-09 更新于 2026-06-10 分类于 Web安全

CSRF（Cross-site request forgery）跨站请求伪造：攻击者诱导受害者进入第三方网站，在第三方网站中，向被攻击网站发送跨站请求。利用受害者在被攻击网站已经获取的注册凭证，绕过后台的用户验证，达到冒充用户对被攻击的网站执行某项操作的目的。
//那现实中的例子来举的话，像跟在别人后面进学校
一个典型的CSRF攻击有着如下的流程：

  • 受害者登录 a.com，并保留了登录凭证（Cookie）。
  • 攻击者引诱受害者访问了 b.com。
  • b.com 向 a.com 发送了一个请求：a.com/act=xx。浏览器会默认携带 a.com 的 Cookie。（从来没有获得过这个Cookie，这是对方的浏览器自动发送的）
  • a.com 接收到请求后，对请求进行验证，并确认是受害者的凭证，误以为是受害者自己发送的请求。
  • a.com 以受害者的名义执行了 act=xx。
  • 攻击完成，攻击者在受害者不知情的情况下，冒充受害者，让 a.com 执行了自己定义的操作。

这个第三方网站就是自己配置的恶意网站。

差不多就是这样的。也就是第三方网站像目标网站发送请求。
把自己想干的事情通过参数穿过去就好了。
