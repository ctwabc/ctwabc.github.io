csrf

CSRF（Cross-site request forgery）跨站请求伪造：攻击者诱导受害者进入第三方网站，在第三方网站中，向被攻击网站发送跨站请求。利用受害者在被攻击网站已经获取的注册凭证，绕过后台的用户验证，达到冒充用户对被攻击的网站执行某项操作的目的。
//那现实中的例子来举的话，像跟在别人后面进学校
一个典型的CSRF攻击有着如下的流程：

  • 受害者登录 a.com，并保留了登录凭证（Cookie）。
  • 攻击者引诱受害者访问了 b.com。
  • b.com 向 a.com 发送了一个请求：a.com/act=xx。浏览器会默认携带 a.com 的 Cookie。（从来没有获得过这个Cookie，这是对方的浏览器自动发送的）
  • a.com 接收到请求后，对请求进行验证，并确认是受害者的凭证，误以为是受害者自己发送的请求。
  • a.com 以受害者的名义执行了 act=xx。
  • 攻击完成，攻击者在受害者不知情的情况下，冒充受害者，让 a.com 执行了自己定义的操作。

这个第三方网站就是自己配置的恶意网站。

把自己想干的事情通过参数穿过去就好了。

普通的 HTML

表单提交时，默认的数据格式是 application/x-www-form-urlencoded（即 key1=value1&key2=value2）。如果你在表单里老老实实写 ，后端一收到，发现不是标准的 JSON 格式，就会直接报 415 Unsupported Media Type 错误并拒绝处理。
也就是说上面的input标签里的会变成name=WebGoat，很显然这不是我们想要。
但是，我们可以调整一下我们的数据。举个例子:

{“name”: “WebGoat”, “email”: “webgoat@webgoat.org“, “content”: “WebGoat is the best!!”, “ignore_me”:”这是一部分
test”}这是一部分
通过=拼接
{“name”: “WebGoat”, “email”: “webgoat@webgoat.org“, “content”: “WebGoat is the best!!”, “ignore_me”:”=test”}
可以看到我们传过去的数据成为json格式了
